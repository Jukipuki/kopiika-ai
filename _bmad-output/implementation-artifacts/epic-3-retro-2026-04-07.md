## Challenges Identified

### 1. Architecture Doc Drifted from Implementation (4+ places)
The `architecture.md` was overridden by `epics.md` multiple times during the epic:
- Corpus path: architecture said `backend/app/rag/content/`, reality is `backend/data/rag-corpus/`
- Embedding model: architecture specified BGE-M3 (1024-dim), reality is OpenAI `text-embedding-3-small` (1536-dim)
- API endpoint: architecture said `/teaching-feed`, epics said `/insights`
- Filename convention: architecture used `_uk.md`/`_en.md` suffixes, reality uses `en/`/`uk/` subdirectories

**Impact:** Architecture doc currently misleads anyone reading it to understand the system.

### 2. SSE Progress Messages Hardcoded on Frontend
`ProgressiveLoadingState.tsx` maps hardcoded step names to copy strings. Adding any new pipeline agent (Epic 4+) falls through to "AI is still thinking..." silently. Backend should own human-readable messages; frontend should render `data.message` directly.

### 3. Locale/Language Switching Doesn't Regenerate Cards
Insight cards are generated at pipeline time with the user's locale at that moment and persisted. Changing language in Settings does not regenerate cards — they stay in the original language. Fundamental UX gap, needs design decision.

### 4. 'Other' Category Cards in Teaching Feed
Low-confidence / unrecognized transactions land in `category='other'`. Insight cards for 'other' are low-value but suppressing them completely risks discarding significant spending data.

**Agreed approach:** Threshold-based logic:
- 'Other' < threshold of total spend → filter silently (no card generated)
- 'Other' ≥ threshold → second LLM re-categorization pass → user clarification for what remains

### 5. Card Visual Hierarchy Issue
Key metric text (large, bold, blue) visually dominates the headline. Eye lands on the metric before understanding its meaning. Screenshot showed: headline in small text, key metric in large blue bold text — semantically backwards.

### 6. Redirect Timing — ProgressiveLoadingState Activates but Pipeline Already Running
User is redirected to Teaching Feed before pipeline completes (by design from Story 3.7 review fix H3). Progressive loading state does show, but as more pipeline stages are added, the current hardcoded step mapping becomes misleading. Related to issue #2.

### 7. Key Metric Content Quality
Some generated key metric values are too dense (e.g., "₴87,582.04 (25.9% of total) vs. ₴213,238.50 finance allocation"). Needs prompt refinement in Education Agent.

---

## Key Insights & Learnings

1. **Architecture doc is not the source of truth** — epics file overrode it 4+ times. Keep architecture in sync with implementation or teams will build on wrong assumptions.
2. **Pipeline coupling is a recurring risk** — SSE step names hardcoded on frontend, locale frozen at generation time. Design new pipeline stages with decoupling as a first-class concern.
3. **Data quality propagates all the way to UX** — 'other' category transactions produce low-value cards. Categorization quality investment pays dividends at every layer above it.
4. **Defer infrastructure migration until the feature that requires it** — Bedrock migration is the right long-term move but the trigger is the chat feature, not Epic 4. The `llm.py` abstraction makes late migration safe.
5. **Real-data testing is essential before quality judgements** — insight quality looks sparse with limited data; a full Monobank month upload should be done before committing to Epic 4 quality baseline.
